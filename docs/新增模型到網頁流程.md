# 將新模型放到網頁並讓前端可選、執行、回傳結果 — 完整清單

以下以 **Moondream（含 moondream3-preview）** 為例，條列「從載好模型到網頁可選、可跑、有結果」需要**新增**與**修改**的檔案，以及每個檔案的用途。下次要接其他模型時，可依此對照調整。

---

## 一、整體流程概覽

1. **後端**：提供「載入模型 + 用你們的 prompt 做推論」的邏輯，並在分析 API 裡多一個分支（例如 `model_type=moondream`）。
2. **前端**：在設定區多了「Model Type」與「Model Version」選項，送出時把 `model_type`、`qwen_model`（或等同的版本參數）與 prompt 一併傳給後端。
3. **Docker/環境**：模型目錄、權重、HF cache、YOLO 權重等透過環境變數與 volume 對應到容器內路徑。
4. **離線/依賴**：若模型會用到 Hub tokenizer 或需額外套件，需補下載腳本或 `requirements.txt`。

---

## 二、後端（Backend）檔案

### 1. **新增**：`backend/src/utils/moondream_utils.py`

| 項目 | 說明 |
|------|------|
| **用途** | Moondream 專用：解析模型目錄路徑、載入模型、對單一影片片段做「事件偵測 + 摘要」推論，回傳與 Qwen 相同格式的 `(frame_obj, summary_txt)`。 |
| **主要內容** | 

• `MOONDREAM_MODELS_BASE`：模型根目錄（支援 `MOONDREAM_MODELS_DIR` 環境變數）<br>
• `get_moondream_model_dir(version)`：依版本名回傳對應模型目錄<br>
• `_sync_trust_remote_code_to_cache(model_dir)`：把模型目錄內 `.py` 同步到 HF cache，避免只複製主檔導致缺 `lora.py` 等<br>
• `_ensure_starmie_tokenizer_local(model_dir)`：若有 `starmie-v1-tokenizer/` 或 `tokenizer/`，patch `Tokenizer.from_pretrained("moondream/starmie-v1")` 改為用 `Tokenizer.from_file(本地 tokenizer.json)`，離線可用<br>
• `load_moondream_model(version, device)`：載入模型（支援 2b 單檔與 3 分片）、可設離線與 `local_files_only=True`<br>
• `_resolve_segment_path(segment_path)`：把片段路徑解析成絕對路徑，必要時用專案根或 backend 目錄當基底<br>
• `infer_segment_moondream(...)`：採樣影格 → 事件偵測（每格）→ 合併事件 → 摘要，回傳 `(frame_obj, summary_txt)` |

| **下次接新模型可做** | 若新模型也是「本地目錄 + trust_remote_code + 自訂 tokenizer」，可複製此檔改為 `xxx_utils.py`，並在 `analysis_service` 裡加一個 `model_type == "xxx"` 分支呼叫對應的 `infer_segment_xxx`。 |

---

### 2. **修改**：`backend/src/services/analysis_service.py`

| 項目 | 說明 |
|------|------|
| **用途** | 分析服務：依 `model_type` 決定用 Qwen、Gemini、Moondream 或 YOLO 做 VLM 分析，再跑 YOLO+ReID；統一回傳格式。 |
| **修改點** | 

• 在 `analyze_segment` 的「1. 執行 VLM 分析」區塊中，新增 `elif req.model_type == "moondream":`<br>
• 該分支內：`try` 裡 import `infer_segment_moondream`、解析 `qwen_model` 當 Moondream 版本、呼叫 `infer_segment_moondream`、用 `_normalize_vlm_output` 得到 `frame_analysis`、寫入 `result["parsed"]`（含 `frame_analysis` 與 `summary_independent`）、設 `result["success"]` / `result["error"]`<br>
• `except` 時仍寫入與 Qwen 相同結構的 `result["parsed"]`（預設事件 + 空摘要），避免前端拿到 `parsed: {}`<br>
• YOLO 失敗時：改為「附加」錯誤到 `result["error"]`，不覆蓋 VLM 的錯誤訊息 |

| **下次接新模型可做** | 再加一個 `elif req.model_type == "新模型名":`，呼叫新模型的 `infer_segment_xxx`，同樣寫入 `result["parsed"]` 與 `result["success"]`。 |

## 七、下次「又載好一個新模型、要放到網頁」時 — 建議步驟（條列）

1. **後端**
   - 新增 `backend/src/utils/新模型_utils.py`（仿 `moondream_utils.py`）：路徑解析、載入、`infer_segment_新模型(...)`，回傳 `(frame_obj, summary_txt)` 與既有格式一致。
   - 在 `analysis_service.py` 的 `analyze_segment` 裡加 `elif req.model_type == "新模型":`，呼叫上述函式並寫入 `result["parsed"]`、`result["success"]`、`result["error"]`。
   - 若新模型需要額外套件，在 `requirements.txt` 補上；若需獨立權重路徑，在 `model_loader.py` 或專用 utils 裡加環境變數/路徑邏輯。
   - 若 API 需要新參數（例如「新模型版本」），在 `video_analysis.py` 的 Form 與 `run_full_pipeline` 參數中加上並傳入 `analyze_segment` 的 req。

2. **前端**
   - 在 `ModelConfig.js` 的 Model Type 加一個選項（如 `value="新模型"`），並在下方加「Model Version」選單（若有），綁定到會送給後端的欄位（例如仍用 `qwenModel` 或新欄位）。
   - 在 `App.js` 的 `formData` 組裝條件中，加上 `modelType === "新模型"`，確保會送出版本、`frames_per_segment`、`event_detection_prompt`、`summary_prompt` 等。

3. **Docker / 環境**
   - 在 `docker-compose.yml` 的 backend 區塊中，若有新模型目錄或權重路徑，加對應的環境變數與 volume（例如 `XXX_MODELS_DIR`、`./models:/app/models` 已存在則不必改）。

4. **離線 / tokenizer**
   - 若新模型有「必須從 Hub 取得的 tokenizer」，寫一個 `scripts/download_xxx_tokenizer.py`，並在說明文件寫何時執行、目錄會放在哪裡；在對應 utils 裡做 patch 或 `from_file` 讓離線可用。

5. **驗證**
   - 重啟 backend（必要時 `docker compose build backend` 再 `up -d backend --force-recreate`）。
   - 前端選新 Model Type / Model Version，上傳影片、使用既有 prompt，確認回傳裡 `parsed.frame_analysis`、`summary_independent` 有內容且格式與 Qwen 一致。

## 九、模型目錄必備結構：以 moondream3-preview 為例與自建模型條件

若你要**自己建立一個新模型**（或 fork 既有模型），讓它能像 Moondream 一樣接到平台，模型目錄內需要具備下列檔案與介面。以下先以 **moondream3-preview** 為例，逐檔說明「平台實際用到了什麼、做什麼」，再整理「自建模型最少要有什麼」。

### 9.1 moondream3-preview 目錄內各檔案 — 我們取了哪些、用途是什麼

| 檔案／目錄 | 平台是否使用 | 用途說明 |
|------------|--------------|----------|
| **config.json** | ✅ 必用 | Hugging Face 載入時讀取。內含 `model_type`、`auto_map`（指向 `hf_moondream.HfConfig`、`hf_moondream.HfMoondream`）、`architectures` 等。`from_pretrained(目錄)` 會依此找到要載入的 Python 類別與設定。 |
| **hf_moondream.py** | ✅ 必用 | 對外介面：定義 `HfConfig`、`HfMoondream`（繼承 `PreTrainedModel`），並把內部 `MoondreamModel` 包成 HF 慣例。平台透過 `AutoModelForCausalLM.from_pretrained(目錄, trust_remote_code=True)` 載入的就是這個類別；載入後的物件需提供 **`.query(image=..., question=...)`**，回傳帶有 **`"answer"`** 的 dict。 |
| **moondream.py** | ✅ 必用 | 實際模型與推理邏輯：`MoondreamModel`、`encode_image`、`query`、`caption` 等。內部會 `Tokenizer.from_pretrained("moondream/starmie-v1")`，平台用 patch 改為從本地 `tokenizer.json` 載入。 |
| **config.py** | ✅ 必用 | 定義 `MoondreamConfig` 等設定結構，供 `hf_moondream`、`moondream` 引用。 |
| **lora.py** | ✅ 必用 | 被 `moondream.py` import（例如 `variant_state_dict`）。若未同步到 HF cache，執行時會報錯缺檔，故平台會把此檔一併複製到 cache。 |
| **vision.py** | ✅ 必用 | 視覺編碼器相關，被 `moondream.py` import。同上，需在目錄內且會被同步到 cache。 |
| **text.py** | ✅ 必用 | 文字／LLM 部分，被 `moondream.py` import。 |
| **region.py** | ✅ 必用 | 座標／區域解碼等，被 `moondream.py` import。 |
| **image_crops.py** | ✅ 必用 | 影像裁切與還原，被 `moondream.py` import。 |
| **layers.py** | ✅ 必用 | 自訂層（如 `QuantizedLinear`），被 `moondream.py` import。 |
| **utils.py** | ✅ 必用 | 工具函式，被 `moondream.py` import。 |
| **rope.py** | ✅ 必用 | 被其他模組引用。 |
| **model.safetensors.index.json** | ✅ 必用 | 分片權重索引；`from_pretrained` 會依此載入對應的 safetensors 檔。 |
| **model-00001-of-00004.safetensors** 等 | ✅ 必用 | 實際權重分片。 |
| **starmie-v1-tokenizer/**（含 tokenizer.json） | ✅ 必用（離線時） | 平台會偵測此目錄，並 patch `Tokenizer.from_pretrained("moondream/starmie-v1")` 改為用 **`Tokenizer.from_file(本地 tokenizer.json)`**，才能離線載入。 |
| .gitattributes、README.md、LICENSE.md、*.png | ❌ 未用 | 不影響載入與推論；可視需要保留或刪除。 |

**平台實際呼叫方式（摘要）**  
- 載入：`AutoModelForCausalLM.from_pretrained(模型目錄, trust_remote_code=True, local_files_only=True, ...)`  
- 推論：對每張影格與摘要各呼叫一次 **`model.query(image=PIL.Image, question=str)`**，並預期回傳 **`dict` 且含鍵 `"answer"`**（或可轉成字串的內容），平台再從 `answer` 抽 JSON 或摘要文字。

### 9.1.1 `moondream_utils.py` 讀取了 moondream3-preview 的哪些檔案、做了什麼

以下依「程式碼流程」列出：**moondream_utils.py** 會碰到 moondream3-preview 目錄裡的哪些東西、以及各自用途。  
（「讀取」包含：檢查存在、複製到 cache、或把路徑交給 transformers/tokenizers 去讀。）

| 順序 | 模型目錄內路徑 | 在 moondream_utils 裡怎麼用 | 做了什麼 |
|------|----------------|-----------------------------|----------|
| 1 | **整個目錄** `models/moondream3-preview/` | `get_moondream_model_dir(version)` 組出 `MOONDREAM_MODELS_BASE / "moondream3-preview"`，並 `path.is_dir()` | 只檢查「目錄存在」；不存在就拋錯，不讀檔內容。 |
| 2 | **starmie-v1-tokenizer/tokenizer.json** | `_ensure_starmie_tokenizer_local(model_dir)` 檢查 `(model_dir / "starmie-v1-tokenizer" / "tokenizer.json").exists()`，若存在則 patch `Tokenizer.from_pretrained`，之後模型 init 時會改呼叫 **`Tokenizer.from_file(該路徑)`** | 我們只做「路徑存在檢查」與 patch；**真正讀取 tokenizer.json 的是 tokenizers 套件**（在載入模型、執行到 `moondream.py` 裡的 `Tokenizer.from_pretrained("moondream/starmie-v1")` 時，被我們導向 `from_file(本地 tokenizer.json)`）。 |
| 3 | **所有 .py 與 .json**（例如 config.json, config.py, hf_moondream.py, moondream.py, lora.py, vision.py, text.py, region.py, image_crops.py, layers.py, utils.py, rope.py, model.safetensors.index.json） | `_sync_trust_remote_code_to_cache(model_dir)` 用 `model_dir.iterdir()` 遍歷，篩選 `f.suffix in (".py", ".json") or f.name == "config.py"`，對每個檔案 **`shutil.copy2(f, dest)`** 複製到 HF cache 的 `transformers_modules/moondream3_hyphen_preview/` | **讀取**：以「複製來源」的方式讀取檔案內容並寫入 cache，讓之後 `from_pretrained(目錄)` 時，模型程式碼的 `from .lora import ...` 等能在 cache 裡找到對應 .py，不會缺檔。**不**在 moondream_utils 裡解析內容。 |
| 4 | **config.json** | 未在 moondream_utils 裡直接 `open`；把 **目錄路徑** 傳給 **`AutoModelForCausalLM.from_pretrained(model_dir_str, ...)`** | **由 transformers 讀取**：`from_pretrained` 會讀目錄內的 config.json，取得 `model_type`、`auto_map` 等，決定要載入哪個 Python 類別與設定。 |
| 5 | **model.safetensors.index.json** + **model-00001-of-00004.safetensors** 等分片 | 同上，只傳目錄路徑給 `from_pretrained` | **由 transformers 讀取**：依 index 載入對應的 .safetensors 分片，填進模型權重。若為單檔版（如 2b）則讀 **model.safetensors**。 |
| 6 | **hf_moondream.py、moondream.py 及所有被 import 的 .py** | 已透過步驟 3 複製到 cache；`from_pretrained` 會依 config.json 的 `auto_map` 從 **cache** 載入對應模組並執行 | **由 transformers 在 cache 裡載入**：執行 HfMoondream、MoondreamModel 等；模型內部再 `from .lora`、`from .vision` 等，都是從 cache 目錄讀。 |

**總結（誰讀了什麼）**

- **moondream_utils.py 自己**：  
  - 只做「目錄存在」檢查、**starmie-v1-tokenizer/tokenizer.json 存在**檢查、以及把 **所有 .py / .json 複製到 HF cache**（讀取檔案內容僅用於複製，不解析）。  
- **tokenizers 套件**：在模型 init 時經我們 patch 後，**讀取 `starmie-v1-tokenizer/tokenizer.json`**。  
- **transformers（from_pretrained）**：在我們傳入的 **模型目錄** 裡讀 **config.json** 與 **權重檔**（model.safetensors 或 index + 分片）；在 **cache** 裡載入並執行我們複製過去的 **.py**（hf_moondream、moondream、lora、vision、text 等）。

因此：**moondream_utils.py 沒有「直接 open 並解析」moondream3-preview 裡的 config 或權重**，而是「準備路徑與 cache、再交給 transformers / tokenizers 去讀」。

### 9.2 自建模型要接到平台 — 必備檔案與介面

若要有一個**自己創建的**新模型，用和目前「後續流程不變」的方式接上平台（同一套前端、同一支分析 API、同一種回傳格式），模型目錄內**至少要具備**以下內容，並滿足介面約定。

#### （1）必備檔案與內容

| 項目 | 說明 |
|------|------|
| **config.json** | • 內含 `model_type`（自訂字串，例如 `"my_vlm"`）<br>• `auto_map` 指向你提供的 Config 與 Model 類別（例如 `"AutoConfig": "hf_my.HfConfig"`, `"AutoModelForCausalLM": "hf_my.HfModel"`）<br>• 其他 HF 或自訂欄位（如 `torch_dtype`、自訂 config 等） |
| **HF 包裝檔（例如 hf_my.py）** | • 類別繼承 `PreTrainedModel`（或 HF 慣例），並在 `auto_map` 中註冊<br>• 載入後物件必須提供 **`.query(image=PIL.Image, question=str)`**，回傳 **`dict`，且至少含鍵 `"answer"`**（字串）；平台會用 `out.get("answer", "")` 取文字做事件解析與摘要 |
| **實際模型檔（例如 my_model.py）** | • 實作模型與 `query` 邏輯（可被 HF 包裝檔 import）<br>• 若內部有 `from .xxx import ...`，則 **所有被 import 的 .py 都要放在同一目錄**（vision、text、lora、utils 等），否則 `trust_remote_code` 複製到 cache 時會缺檔 |
| **config.py（或同等設定）** | • 供 HF 包裝與模型使用；若 config.json 裡有自訂結構，需有對應的類別或讀取方式 |
| **權重** | • 單檔：**model.safetensors**<br>• 或分片：**model.safetensors.index.json** + **model-00001-of-xxx.safetensors** 等；檔名與 index 內容需一致 |
| **Tokenizer（若模型會用）** | • **做法 A**：模型內不呼叫 Hub，改為直接讀取「模型目錄下的子目錄」（例如 `tokenizer/`）裡的 **tokenizer.json**，平台可再依需要 patch 成 `Tokenizer.from_file(路徑)`<br>• **做法 B**：模型內仍呼叫 `Tokenizer.from_pretrained("some/repo")`，則需在平台側提供「本地 tokenizer 目錄」並做 patch（同 Moondream starmie-v1），否則離線會失敗 |

#### （2）平台對「載入後模型」的介面假設（與 Moondream 一致）

- **`model.query(image= PIL.Image, question= str)`**  
  - 回傳：**`dict` 且含 `"answer"`**（字串），或可被當成字串使用的物件；平台會做 `out.get("answer", "")` 或 `str(out)`。  
- 平台會對「事件偵測」的 answer 做 JSON 解析（抽 `events`），對「摘要」的 answer 直接當成 `summary_independent` 文字。  
- 若你的模型介面不同（例如方法名不是 `query`、或回傳格式不同），則需要在 **backend 的 `xxx_utils.py`** 裡做一層轉換（例如在 `infer_segment_xxx` 裡呼叫你的方法並轉成 `{"answer": ...}`）。

#### （3）自建模型目錄建議結構（最少）

```
models/
  我的模型名/
    config.json              # 必備，含 model_type、auto_map
    hf_我的模型.py            # 必備，HF 包裝，暴露 .query(image, question) -> {"answer": str}
    我的模型.py               # 必備，實際模型與推理
    config.py                # 若 config 有自訂結構則必備
    (其他被 import 的 .py)    # 所有在「同一目錄」內被 import 的都要有
    model.safetensors        # 單檔權重
    # 或
    model.safetensors.index.json
    model-00001-of-xxx.safetensors
    ...
    tokenizer/               # 若需離線 tokenizer
      tokenizer.json
      (其他 tokenizer 檔)
```

#### （4）接上平台時你要多做的事（流程不變的前提）

- 在 **backend** 新增對應的 **`xxx_utils.py`**（路徑解析、載入、**呼叫你的 `model.query(image, question)`**、組 `(frame_obj, summary_txt)`）。  
- 在 **analysis_service** 加 **`model_type == "我的模型"`** 分支，呼叫該 utils 的推論函式。  
- 若你的模型有「依名稱從 Hub 載入 tokenizer」，要麼改為讀本地目錄，要麼在 utils 裡做 tokenizer patch（同 starmie-v1）。  
- **config.json 的 `auto_map`** 與目錄內檔名、類別名要一致，且載入後的類別實例要提供 **`.query(image=..., question=...)` 且回傳含 `"answer"` 的 dict**，平台端就不必改前端或 API 契約。

---

以上涵蓋「從載好模型到網頁可選、可跑、有結果」需要動到的所有檔案與其用途，以及**模型目錄本身**要具備的檔案與介面；下次接新模型或自建模型時，可依此對照修改與新增。
