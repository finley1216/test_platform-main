# RTSP 錄影 + API Pipeline：測試與觀察指南

## 一、怎麼執行（建議順序）

### 1. 後端與 Ollama 先起來

```bash
cd /mnt/10THDD/M133040024/SSD/ASE/test_platform-main
docker compose up -d    # 或至少 backend + ollama
```

確認 API 可連、Ollama 有跑（例如 `curl http://localhost:11434/api/tags` 或你實際的 BACKEND_URL）。

### 2. 開錄影（RTSP Recorder）

**先改設定**：`rtsp-recorder/config/recorder-config.yaml` 裡把 `recording.enabled` 改為 `true`，才會真的錄。

```bash
cd /mnt/10THDD/M133040024/SSD/ASE/test_platform-main/rtsp-recorder
export PROJECT_ROOT="$PWD"   # 錄影寫到 ./video/
python src/main.py
```

- 會**一直跑**，直到你 Ctrl+C 或達到 `max_duration`（預設 3600 秒）。
- 每 **10 秒**寫一支檔（`segment_duration: 10`），檔名如：`video/Test_Site-Ch1-2025-02-22_14-30-00.avi`。
- **切割出來的影片**就在 **`rtsp-recorder/video/`**（與 `output.directory: ${PROJECT_ROOT}/video` 一致）。

### 3. 開 API Pipeline（等切割好的影片送 API）

**另開一個終端**，同專案根目錄：

```bash
cd /mnt/10THDD/M133040024/SSD/ASE/test_platform-main
export BACKEND_URL="http://140.117.176.88:3000/api"   # 改成你的後端
export MY_API_KEY="你的API_KEY"

# 只跑一輪（掃一次 video/，有未處理的就送 API）
python rtsp-recorder/run_api_pipeline.py --once

# 或持續監看：每 30 秒掃一次，4 個並行
python rtsp-recorder/run_api_pipeline.py --workers 4
```

- Pipeline **不會主動切影片**，只會掃「已經在 video/ 裡的檔」並上傳。
- 要「同步」= 先跑 recorder 讓 video/ 有檔，再跑 pipeline；pipeline 每 `--interval` 秒掃一次，有**新檔且超過 `--min-age` 秒**才送 API（避免寫檔還沒寫完就送）。

## 二、Ollama 秒數怎麼看

每次後端呼叫 Ollama（VLM），Ollama 容器會打一筆 log，例如：

```text
ollama-1  | [GIN] 2026/02/22 - 12:35:15 | 200 | 2.030608408s | 172.19.0.6 | POST "/api/chat"
```

- **中間那個 `2.030608408s`** 就是**這一通 Ollama 請求的耗時**（可當成 VLM 秒數）。

看即時 log：

```bash
cd /mnt/10THDD/M133040024/SSD/ASE/test_platform-main
docker compose logs -f ollama
```

只看最近幾筆也可用：

```bash
docker compose logs --tail 50 ollama
```

## 三、每段執行時間在哪看

### 3.1 API 回應裡（單次上傳）

呼叫 `POST /v1/segment_pipeline_multipart` 後，回應 JSON 裡有：

| 欄位 | 意義 |
|------|------|
| `total_time_sec` | 從收到請求到回傳的**整段時間**（含收檔、切割、分析、寫 DB）。 |
| `process_time_sec` | **只算「分析」**：從 `run_full_pipeline` 開始到結束（不含收檔、prepare_segments）。 |
| `results` | 每個片段的結果陣列。 |
| `results[i].time_sec` | **該片段**從開始分析到結束的秒數（VLM + YOLO + ReID 等）。 |

Pipeline 終端也會印出每支的總時間，例如：

```text
  [OK] Test_Site-Ch1 / Test_Site-Ch1-2025-02-22_14-30-00  total_time_sec=8.5
```

### 3.2 後端 log（整體與錯誤）

後端會打：

```text
--- [Pipeline Success] Time: 8.52s, Mem Delta: 12.3MB ---
```

這是**整次請求**的耗時。要看後端 log（依你啟動方式）：

```bash
docker compose logs -f backend
# 或
# 若用 uvicorn 直接跑：看跑 backend 的那個終端
```

### 3.3 存成 JSON 的完整結果（含每段 time_sec）

若 API 有開 `save_json: True`（pipeline 預設會送），後端會把整次回應寫到 **segment 目錄** 的 JSON，例如：

- 路徑在回應的 `save_path`，或到後端設定的 `SEGMENT_DIR` / 專案下的 `backend/segment/<stem>/` 找對應的 `<stem>.json`。
- 裡面有 `results[].time_sec`、`process_time_sec`、`total_time_sec`，可事後查每段與整次時間。

## 四、RTSP 會跑多久、怎麼確認「API 在等切割的影片」」

### RTSP 跑多久

- 由 **`config/recorder-config.yaml`** 決定：
  - `recording.max_duration: 3600` → 單路最長錄 **3600 秒（1 小時）**，到就停。
  - 沒設或設很大 → 會一直錄到 **手動 Ctrl+C**。
- 每 **10 秒**寫一支檔（`segment_duration: 10`）。

### 怎麼確認「已經在同步（API 在等切割的影片）」

1. **錄影有在寫檔**  
   - 看 `video/` 有沒有新檔、時間戳是否每約 10 秒增加。  
   - 路徑：`<PROJECT_ROOT>/video/` 或 `test_platform-main/rtsp-recorder/video/`（依你的 `output.directory`）。

2. **Pipeline 有掃到並送 API**  
   - 跑 pipeline 的終端會印「待處理: N 支」、以及 `[OK] 路名/檔名 total_time_sec=...`。  
   - 若一直「待處理: 0」，表示：  
     - video/ 路徑不對（改 `VIDEO_LIB_DIR` 或讓 recorder 寫到 pipeline 會掃的那個目錄），或  
     - 檔案還沒超過 `--min-age`（預設 2 秒），或  
     - 該檔已經在 `_api_pipeline_processed.json` 裡被標成已送過。

3. **Ollama 有被呼叫**  
   - `docker compose logs -f ollama` 看到 `POST "/api/chat"` 且 200，就代表後端有在叫 Ollama，代表 API 有在處理你上傳的影片。

## 五、用的模型是什麼

- **Pipeline 送給 API 的**：在 `run_api_pipeline.py` 裡是 **`qwen_model="qwen2.5vl:latest"`**，也可用環境變數覆蓋：
  - `OLLAMA_LLM_MODEL=qwen2.5vl:7b python rtsp-recorder/run_api_pipeline.py --once`
- **後端預設**：`video_analysis.py` 的 Form 預設是 `qwen_model: str = Form("qwen3-vl:8b")`，但 **pipeline 送的表單會帶 `qwen_model`**，所以實際用的是 pipeline 傳的 **qwen2.5vl:latest**（除非你在程式或 env 裡改掉）。
- 要確認：看後端 log 的 `--- [Pipeline Start] Model: qwen, Qwen: qwen2.5vl:latest ---`，或看 API 回應裡的 `model_type` / 後端設定。

## 六、切割出來的影片在哪裡

- **「切割」在這裡 = RTSP 錄影寫出的每段檔**，不是後端再切一次。
- 路徑由 **recorder 的 `config/recorder-config.yaml`** 的 `output.directory` 決定：
  - 預設是 **`${PROJECT_ROOT}/video`**。
  - 若你在 **`rtsp-recorder/`** 目錄下跑、且 `export PROJECT_ROOT="$PWD"`，錄影會寫到：
    - **`test_platform-main/rtsp-recorder/video/`**
  - 檔名格式：**`<location>-<camera>-<YYYY-MM-DD_HH-MM-SS>.avi`**  
    例：`Test_Site-Ch1-2025-02-22_14-30-00.avi`。

若你有用 **VIDEO_LIB_DIR**（或 pipeline 的預設搜尋順序），pipeline 會依同一套規則去找「錄影目錄」；上面路徑就是「切割出來的影片」所在目錄。

## 七、快速檢查清單

| 想確認的事 | 怎麼看 |
|------------|--------|
| Ollama 有被呼叫、每通幾秒 | `docker compose logs -f ollama` → 看 `[GIN] ... \| XXXs \| POST "/api/chat"` |
| 每段分析幾秒 | API 回應的 `results[i].time_sec`，或存下來的 JSON `save_path` |
| 整次請求幾秒 | API 回應的 `total_time_sec`、`process_time_sec`；後端 log 「Pipeline Success Time: Xs」 |
| 用的模型 | 後端 log 「Qwen: qwen2.5vl:latest」或 API 回應 / pipeline 傳的 `qwen_model` |
| 切割影片在哪 | `PROJECT_ROOT/video/` 或 `rtsp-recorder/video/`，檔名 `xxx-YYYY-MM-DD_HH-MM-SS.avi` |
| RTSP 跑多久 | `recorder-config.yaml` 的 `max_duration`，或手動 Ctrl+C |
| 是否同步（API 在吃切割檔） | video/ 有新檔 + pipeline 印「待處理: N」且出現 `[OK]` + ollama log 有 POST /api/chat |
